# 猫维斯（Catvis）完整需求文档

## 1. 文档目的

本文件用于定义 `猫维斯（Catvis）` 的产品目标、边界、核心场景、功能需求、非功能需求、验收标准和阶段规划，作为后续设计、实现和排期依据。

配套文档：

- [CATVIS_ARCHITECTURE.md](./CATVIS_ARCHITECTURE.md)
- [CATVIS_TECH_BREAKDOWN.md](./CATVIS_TECH_BREAKDOWN.md)

## 2. 项目概述

`猫维斯（Catvis）` 是一个桌面常驻型 AI 助手形态。它不以聊天框为主，而是以一只持续存在的动态猫猫作为可见界面，让用户通过麦克风进行输入，并通过屏幕表情、字幕和音响播报获得回应。

产品底座不是新造一套 Agent，而是复用现有稳定的 `XiaoBa Runtime`。换句话说，`猫维斯` 的核心不是“新的脑子”，而是“给 XiaoBa 一个可见、可听、可打断的身体”。

## 3. 背景与机会

当前 `XiaoBa Runtime` 已经在这些形态中具备稳定性：

- CLI 对话
- 平台消息机器人
- Dashboard 管理界面
- 角色、skills、tools 的统一 runtime

但仍然存在明显缺口：

- 缺少持续在线的桌面存在感
- 交互入口偏工具化，仍然更像“调用一个系统”
- 用户无法通过界面立即判断它是否空闲、是否在思考、是否在输出
- 语音输入和语音输出尚未成为一等交互方式

这就给 `猫维斯` 留出了清晰机会：

- 保留 `XiaoBa Runtime` 的稳定核心
- 把输入从“键盘输入文本”扩展为“麦克风输入语音”
- 把输出从“send_text / 屏幕文本”扩展为“语音播报 + 字幕 + 视觉状态”
- 让电脑本身成为第一代硬件载体

## 4. 产品定义

一句话定义：

`猫维斯 = 一个以 XiaoBa Runtime 为大脑、以本机麦克风和音响为感官与发声器官、以动态猫猫为可视界面的桌面常驻 Agent。`

它不是：

- 一个宠物挂件
- 一个只会播放动画的皮肤
- 一个独立于 `XiaoBa Runtime` 的新助手系统

它是：

- `XiaoBa Runtime` 的 embodied desktop shell
- 语音化、状态可视化、常驻化的交互形态

## 5. 产品愿景

把“稳定的助手”变成“有身体感的存在”。

用户不再只是打开一个聊天软件，而是在电脑上拥有一个一直在线、能听、能想、能说、能表现状态的猫系助手。长期目标不是让它更花哨，而是让它越来越像一个可以持续协作的桌面存在。

## 6. 目标与非目标

### 6.1 产品目标

第一阶段要达成：

1. 用户可以直接对着电脑说话调用 XiaoBa
2. 猫猫在屏幕上持续展示系统运行状态
3. 回复同时通过语音和字幕输出
4. 用户可以打断当前播报，进入下一轮交互
5. 整个体验形成“电脑就是助手身体”的感受

### 6.2 技术目标

第一阶段技术上要达成：

1. 不重写 `XiaoBa Runtime`
2. 用适配器把语音输入转换成标准文本 turn
3. 用适配器把 runtime 文本输出转换成 TTS 播报
4. 建立稳定的 `闲置 / 推理 / 输出` 三态映射
5. 关键链路全程可观测、可记录、可回放

### 6.3 非目标

第一阶段明确不追求：

1. 外接硬件接入
2. 复杂情绪引擎
3. 全自动长期监听
4. 家庭多设备联动
5. 高度拟真的 3D 角色动画系统
6. 纯离线语音模型闭环
7. 系统级复杂自动化编排

## 7. 核心价值主张

对用户的核心价值不是“功能更多”，而是“交互方式更自然、状态更可感知”。

具体体现在：

1. 更低的交互门槛：可以直接说话，而不是先切到聊天窗口再打字
2. 更强的状态可见性：能立刻知道它有没有听到、有没有在想、有没有在说
3. 更强的存在感：桌面上始终有一个持续在线的角色，而不是一次性工具
4. 更高的连续性：仍然复用现有 `XiaoBa Runtime` 的能力，不牺牲稳定性

## 8. 目标用户

### 8.1 核心用户

- 你自己，以及愿意长时间把 AI 助手挂在桌面的重度用户

### 8.2 用户特征

- 已经习惯使用 `XiaoBa`
- 接受常驻桌面助手
- 希望助手更像一个存在，而不是一次性工具
- 对“状态感”和“陪伴感”有要求
- 能容忍第一版以稳定性优先，而不是极致拟人化

### 8.3 用户分层

`P0 用户`

- 深度使用 XiaoBa 的 creator / builder
- 能忍受原型限制
- 愿意提供日志和反馈

`P1 用户`

- 对桌面常驻 AI 感兴趣的重度用户
- 希望更自然的输入方式
- 对 UI 状态一致性要求高

## 9. 典型场景

### 9.1 日常问答

用户打开电脑后，猫猫常驻桌面一角。用户按键或点击开始说话：

- “帮我整理今天要做的事”
- “总结一下我刚才说的这段想法”
- “我下午先做什么最合理”

猫猫进入推理态，随后通过语音和字幕回答。

### 9.2 工作陪伴

用户在写作、做方案、查资料时不想来回切窗口，希望助手一直在旁边：

- 能看见它是不是在工作
- 能知道它是不是在思考
- 能在它说话时快速打断

### 9.3 快速捕捉灵感

用户临时想到一句话或一段结构，不想打开聊天框输入，希望直接开口说：

- “把这个想法整理成一个产品结构”
- “帮我记住这个点，晚上再展开”

这类场景要求更快的唤起速度和更轻的操作负担。

### 9.4 未来扩展场景

后续可以扩展到：

- 日程提醒
- 待办读取
- 主动播报
- 本地工具调用
- 系统通知联动

但这些不属于第一阶段 MVP。

## 10. 产品核心体验原则

### 10.1 先稳定，后华丽

第一阶段必须优先确保：

- 能听到
- 能理解
- 能回答
- 能播报
- 能正确切状态

而不是优先做复杂动作或表情库。

### 10.2 视觉状态必须服务运行状态

猫猫的动画不是装饰，它必须直接反映实际系统状态。

### 10.3 用户必须知道系统现在在干什么

用户不应该出现下面这些疑问：

- 它到底有没有开始听
- 它是不是识别失败了
- 它是不是卡住了
- 它是不是已经说完了

### 10.4 常驻但不侵入

它应该一直在，但不能像直播挂件一样抢焦点、挡内容、过度占据屏幕中心。

## 11. 端到端交互流程

### 11.1 正常流程

1. 用户触发录音
2. 猫猫给出“已开始接收输入”的即时反馈
3. 录音结束后进入推理态
4. STT 产出文本
5. 文本进入 `XiaoBa Runtime`
6. runtime 返回文本回复
7. 系统启动 TTS
8. 猫猫切到输出态
9. 音响播报回答，字幕同步显示
10. 播报结束，回到闲置态

### 11.2 中断流程

1. 系统正在播报
2. 用户再次触发录音
3. 当前播报被中止
4. 上一轮 turn 标记为 interrupted
5. 新一轮输入开始

### 11.3 失败恢复流程

1. 某一环节出错
2. 猫猫展示明确错误反馈
3. 调试面板记录失败原因
4. 系统回到可继续使用的闲置态

## 12. 功能需求

## 12.1 桌面壳与常驻

需求：

- 应用以 Electron 形式运行
- 支持托盘常驻
- 支持猫猫窗口显示、隐藏、恢复
- 支持小窗、置顶或靠边停驻形态
- 启动后自动加载本地服务和猫猫界面

验收：

- 用户启动应用后 10 秒内能看到猫猫
- 关闭主窗口后仍可从托盘恢复
- 不影响已有 dashboard/service 运行逻辑

## 12.2 语音输入

需求：

- 使用系统默认麦克风
- 支持点击说话或按键说话
- 明确显示“正在录音”状态
- 将语音稳定转成文本并送入 runtime

约束：

- MVP 不做唤醒词
- MVP 不做后台持续监听

验收：

- 用户完成一次说话后，系统能在 2 秒内进入明确的“处理中”状态
- 识别文本可在调试界面查看
- 麦克风不可用时有明确错误提示

## 12.3 三态可视化

需求：

- 至少具备 `闲置 / 推理 / 输出` 三个外显状态
- 状态变化必须和真实系统事件一致
- 每个状态都有明显、可感知、低歧义的视觉差异

状态定义：

1. `闲置`：等待交互
2. `推理`：听、转写、思考统一映射到该态
3. `输出`：播报回答、展示字幕、展示结果

验收：

- 用户能在不看日志的情况下判断当前大致状态
- 核心路径中不能出现明显状态错位

## 12.4 语音输出

需求：

- 将 runtime 的文本回答转换成可播放语音
- 通过系统音响播放
- 支持播报中断
- 支持播报结束回调，驱动状态恢复

验收：

- 回答开始播报时，猫猫必须同步进入 `输出` 态
- 用户再次发起输入时，系统可停止当前播报并进入下一轮
- TTS 失败时系统可恢复到闲置态

## 12.5 字幕与文本展示

需求：

- 在播报时同步显示字幕
- 字幕应可读、可跟随输出更新
- 支持显示完整文本或 speakable text

验收：

- 至少有一个清晰可读的字幕区
- 播报内容和字幕内容保持基本一致

## 12.6 Runtime 复用

需求：

- 使用现有 `XiaoBa Runtime` 处理对话
- 不允许为 `猫维斯` 再维护一套独立 agent 内核
- 现有角色、skills、tools 原则上可继续复用

验收：

- voice turn 最终进入标准会话日志
- runtime 输出仍可回流到现有 observability 体系

## 12.7 状态控制与中断

需求：

- 建立清晰的 turn 生命周期
- 能处理中断、失败、取消、超时
- 播报期间新输入优先级高于旧输出

验收：

- 被中断的 turn 有明确状态标记
- 中断后不出现残余音频继续播放
- 中断后 UI 不停留在错误状态

## 12.8 设置与调试

需求：

- 可查看当前 STT/TTS 配置
- 可查看最近 turn 的识别文本、回复文本、状态切换和错误信息
- 可切换输入模式、字幕开关、自动播报开关

验收：

- 至少存在一个调试面板和一个基础设置面板
- 不需要进入终端就能排查主要问题

## 12.9 可观测性

需求：

- 记录语音 turn 的关键日志
- 记录 STT、runtime、TTS、avatar 的时延
- 支持问题回放和后续分析

验收：

- 至少能查看每次 turn 的开始时间、识别文本、回复文本、状态切换
- 失败原因可定位到语音链路、runtime 或渲染层

## 13. 非功能需求

## 13.1 性能

MVP 目标：

- 应用冷启动到可见：`<= 10s`
- 录音结束到进入推理态：`<= 2s`
- runtime 返回后到开始播报：`<= 1.5s`
- 三态切换的视觉延迟：`<= 200ms`

## 13.2 稳定性

目标：

- 连续运行 4 小时不崩溃
- 连续 30 轮对话不出现明显状态错乱
- 发生 STT/TTS 网络失败后系统能自恢复

## 13.3 隐私与权限

要求：

- 明确展示麦克风是否正在工作
- 明确区分“可输入状态”和“未输入状态”
- 默认不做无提示常开监听
- 配置页中说明音频会被发送到所选语音服务

## 13.4 可维护性

要求：

- 语音链路、runtime 适配层、avatar 渲染层尽量模块化
- 配置和 provider 必须可替换
- 必须保留日志，方便回流给 `InspectorCat`

## 13.5 可扩展性

要求：

- 后续可以扩展唤醒词
- 后续可以扩展主动播报
- 后续可以增加更多动作与情绪映射
- 后续可以增加系统级工具联动

## 14. 用户故事

1. 作为一个长期使用 XiaoBa 的用户，我希望直接对电脑说话，而不是切到聊天框输入，这样我能更自然地调用助手。
2. 作为一个桌面常驻助手使用者，我希望能一眼看出它是否在听、是否在想、是否在说，这样我不会不知道系统卡在哪。
3. 作为一个高频用户，我希望它在说话时我可以打断，这样交互才像真实助手，而不是播放器。
4. 作为开发者，我希望保留完整 turn 日志和错误信息，这样我能继续用 `XiaoBa World` 的 loop 去优化它。

## 15. MVP 范围

MVP 只做最小闭环：

1. Electron 常驻桌面
2. 动态猫猫常驻显示
3. 点击/按键说话
4. STT 转写
5. 将语音文本送入 `XiaoBa Runtime`
6. TTS 播报
7. `闲置 / 推理 / 输出` 三态切换
8. 基本字幕区
9. 基本日志与调试页
10. 播报中断能力

## 16. 明确不进 MVP 的内容

以下内容延期：

1. 唤醒词
2. 纯离线语音识别
3. 多角色切换表演
4. 主动任务编排
5. 系统级自动化控制
6. 复杂表情库和骨骼动画
7. 多设备联动
8. 云端用户同步

## 17. 关键交互规则

### 17.1 输入规则

- 默认需要用户显式触发录音
- 录音开始时猫猫必须给出明显反馈
- 录音结束后立即结束输入态，不可长时间悬空

### 17.2 推理规则

- 一旦录音结束，猫猫立即切到推理态
- runtime 未返回前，不允许误回到闲置态
- STT 和 runtime 可以属于内部不同状态，但外显统一为推理态

### 17.3 输出规则

- 只有真正开始播报时才进入输出态
- 输出态应带字幕
- 播报结束后回闲置态

### 17.4 中断规则

- 新输入优先级高于当前播报
- 中断后旧 turn 标记为 interrupted
- 中断后的视觉状态必须即时反映新 turn 开始

## 18. 失败场景要求

系统至少要覆盖以下失败反馈：

1. 麦克风不可用
2. 权限被拒绝
3. STT 失败
4. runtime 超时
5. TTS 失败
6. 网络异常
7. 本地渲染异常

要求：

- 猫猫给出明确错误表情或提示
- 调试面板能看到失败原因
- 系统能回到可继续使用的闲置状态
- 失败不能导致整套应用卡死

## 19. 依赖与前提假设

### 19.1 依赖

- 现有 `XiaoBa-CLI`
- 现有 Electron 壳与 dashboard server
- `cat-orb-demo` 的动态猫猫前端能力
- 至少一个可接入的 STT 服务
- 至少一个可接入的 TTS 服务

### 19.2 前提假设

- `XiaoBa Runtime` 继续作为统一对话内核
- 第一阶段允许使用云端 STT/TTS
- 语音 provider 可替换，不与单一厂商强绑定

## 20. 成功指标

### 20.1 定性指标

如果满足下面四点，就说明 `猫维斯` 方向成立：

1. 用户愿意把它长时间挂在桌面上
2. 用户更愿意“对它说”而不是“打开聊天框打字”
3. 用户能从猫猫状态中感知系统工作进度
4. `XiaoBa Runtime` 的稳定性被放大成一种存在感，而不是被界面稀释

### 20.2 定量指标

MVP 阶段建议追踪：

- 日均语音 turn 数
- 语音 turn 成功率
- STT 成功率
- TTS 成功率
- 中断成功率
- 用户平均单次会话轮数
- 崩溃率和异常恢复率

### V0.1 原型验证版

目标：

- 跑通三态切换
- 跑通语音输入到 TTS 输出
- 证明“电脑就是硬件壳”成立

### V0.2 桌面常驻版

目标：

- 托盘
- 小窗
- 配置项
- 字幕区
- 中断能力
- 基本调试页

### V0.3 Agent Presence 版

目标：

- 更丰富动作表情
- 更好的陪伴感
- 更稳定的主动提示与状态感
- 更多状态与情绪映射

## 22. 发布门槛

MVP 可以对外演示前，至少满足：

1. 从语音输入到语音输出的闭环稳定可复现
2. 三态切换可被用户明显感知
3. 主要错误路径都能自恢复
4. 日志足够支撑下一轮优化

## 23. 当前结论

`猫维斯` 不是一个装饰性宠物挂件，也不是给现有聊天 UI 加个皮肤。

它的本质是：

`把 XiaoBa 从“可调用的助手”变成“有身体、有状态、能听能说、持续在场的桌面 Agent”。`

第一阶段最重要的不是功能堆叠，而是把下面这条链稳定打通：

`用户说话 -> XiaoBa 理解 -> 猫猫表现推理 -> XiaoBa 回答 -> 猫猫开口说话`

只要这条链成立，后面的唤醒词、主动提醒、桌面工具联动都会变成自然扩展，而不是空中楼阁。
